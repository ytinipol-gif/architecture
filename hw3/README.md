## Описание предметной области
Система предназначена для:
- хранения пользователей
- создания событий
- регистрации пользователей на события

В системе используются три основные сущности:
- Пользователь (users)
- Событие (events)
- Участник события (event_participants)

## Структура базы данных
### Таблица users
Хранит информацию о пользователях.

Основные поля:
- id - первичный ключ
- login - уникальный логин
- first_name - Имя
- last_name - Фамилия
- email - уникальный email
- password_hash - пароль

Ограничения:
- login UNIQUE
- email UNIQUE
- NOT NULL на основные поля

### Таблица events
Хранит информацию о событиях.

Основные поля:
- id - первичный ключ
- title - Название
- description - описание
- event_date - дата события
- location - место проведения
- organizer_id - организатор (FK на users)

### Таблица event_participants
Связывает пользователей и события.

Основные поля:
- id - первичный ключ
- event_id - FK на events
- user_id - FK на users
- status - статус регистрации

Ограничения:
- UNIQUE (event_id, user_id)
- CHECK для статуса

## Индексы
В базе данных используются индексы для ускорения запросов:

- users(login) - поиск пользователя по логину  
- users(first_name), users(last_name) - поиск по Имени и Фамилии  
- events(event_date) - поиск событий по дате  
- events(organizer_id) - получение событий организатора  
- event_participants(event_id) - участники события  
- event_participants(user_id) - события пользователя  
Индексы по первичным и уникальным ключам создаются автоматически.

## SQL-файлы
В репозитории присутствуют следующие файлы:

- `schema.sql` - создание таблиц и индексов  
- `data.sql` - тестовые данные  
- `queries.sql` - основные и оптимизированные запросы  
- `optimization.md` - анализ и описание оптимизации запросов  

## Основные операции системы
Реализованы следующие операции:

- создание пользователя  
- поиск пользователя по логину  
- поиск пользователя по имени/фамилии  
- создание события  
- получение списка событий  
- поиск событий по дате  
- регистрация пользователя на событие  
- получение участников события  
- получение событий пользователя  
- отмена регистрации  

Также добавлены аналитические запросы:
- популярные события  
- количество событий у организаторов  
- распределение событий по городам  

## Оптимизация
Оптимизация выполнялась за счёт: добавления индексов по часто используемым полям, уменьшения использования SELECT *, фильтрации данных (например, status = 'registered'), оптимизации JOIN-запросов.

Для анализа использовался EXPLAIN.
Основной прирост производительности дали индексы по:
- login
- event_date
- event_id
- user_id

## Партиционирование
В данной работе партиционирование не реализовано, так как объём данных небольшой.

Однако в реальной системе можно использовать:
- партиционирование таблицы events по дате
- партиционирование таблицы event_participants по дате регистрации (подробнее описано в optimization.md)
