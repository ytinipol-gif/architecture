# Оптимизация запросов sql
## Общая информация

После создания и заполнения реляционной базы данных для системы управления событиями. Были написаны sql-запросы к БД, после чего проведён анализ основных SQL-запросов.

Основная цель оптимизации:
- уменьшить количество полных сканирований таблиц;
- ускорить поиск по часто используемым полям;
- повысить эффективность JOIN-запросов;
- ускорить выборки по дате и связям между сущностями.

В системе используются три основные таблицы:
- users
- events
- event_participants

## Используемые индексы и их назначение

В базе данных были созданы следующие индексы:
1. Уникальный индекс по login (создаётся через UNIQUE)  - используется для быстрого поиска пользователя по логину.
2. Уникальный индекс по email - используется для проверки уникальности при регистрации.
3. Индекс по first_name - ускоряет поиск пользователей по имени.
4. Индекс по last_name - ускоряет поиск пользователей по фамилии.
5. Индекс по event_date - используется для быстрого поиска событий по дате.
6. Индекс по organizer_id - позволяет быстро получать события конкретного пользователя.
7. Индекс по event_id в таблице event_participants - ускоряет получение участников события.
8. Индекс по user_id в таблице event_participants - ускоряет получение событий пользователя.
9. Уникальный индекс (event_id, user_id) - не позволяет зарегистрироваться на одно событие дважды и ускоряет поиск конкретной регистрации.

## Анализ и оптимизация запросов

### 1. Поиск пользователя по логину

Запрос:
SELECT * FROM users WHERE login = 'maria_lemeshchenko';

До оптимизации:
Seq Scan on users

После оптимизации:
Index Scan using users_login_key

Вывод:
Индекс позволяет находить пользователя без полного просмотра таблицы. Если бы индекс по login отсутствовал, PostgreSQL пришлось бы просматривать все строки таблицы users, то есть использовать последовательное сканирование (Seq Scan). После добавления уникального ограничения по login запрос может выполняться через индексный поиск (Index Scan). Это необходимо, если таблица пользователей сбудет содержать большой массив данных.

### 2. Поиск пользователя по маске имени или фамилии

Запрос:
SELECT id, login, first_name, last_name
FROM users
WHERE LOWER(first_name) LIKE '%мар%'
   OR LOWER(last_name) LIKE '%мар%';

Особенность:
Индексы по first_name и last_name используются ограниченно, так как поиск начинается с %.

Вывод:
Этот запрос оптимизируется хуже, чем поиск по логину. Потому, что шаблон начинается с символа %, а значит обычный B-Tree индекс используется ограниченно. Но индексы по first_name и last_name всё равно оправданы, потому что в реальной системе поиск может выполняться и по более простым шаблонам. Здесь так же можно было бы использовать полнотекстовый поиск или триграммные индексы.

### 3. Получение списка событий

Запрос:
SELECT id, title, event_date, location
FROM events
ORDER BY event_date DESC;

Оптимизация:
Используется индекс по event_date.

Вывод:
Индекс помогает ускорить сортировку и выборку. Для такого запроса полезен индекс по event_date, так как он помогает не только при поиске по дате, но и при сортировке событий. Это особенно актуально, если в системе хранится много событий и пользователи регулярно просматривают их список.

### 4. Поиск событий по дате

Запрос:
SELECT * FROM events WHERE event_date = '2026-04-20';

До оптимизации:
Seq Scan on events

После оптимизации:
Index Scan using idx_events_event_date

Вывод:
Индекс значительно ускоряет фильтрацию по дате. До добавления индекса по event_date план выполнения выглядел бы как последовательное сканирование таблицы events. После добавления индекса запрос становится эффективнее, потому что серверу больше не нужно просматривать всю таблицу.

### 5. Регистрация пользователя на событие

Запрос:
INSERT INTO event_participants (event_id, user_id, status)
VALUES (1, 5, 'registered');

Особенность:
Оптимизация достигается за счёт внешних ключей и уникального ограничения (event_id, user_id).

Вывод:
Тут важнее контроль целостности данных, а не скорость запроса.


### 6. Получение участников события

Запрос:
SELECT u.id, u.login
FROM users u
JOIN event_participants ep ON u.id = ep.user_id
WHERE ep.event_id = 1;

До оптимизации:
Seq Scan по event_participants

После оптимизации:
Index Scan по event_id

Вывод:
Индекс значительно ускоряет JOIN. Ключевым является индекс по event_id в таблице event_participants. Сначала база данных быстро выбирает все регистрации нужного события, а затем соединяет их с таблицей users по первичному ключу. Без индекса по event_id пришлось бы каждый раз просматривать всю таблицу регистраций. Дополнительно фильтрация по status = 'registered' позволяет сразу исключить отменённые регистрации и уменьшить объём лишних данных.

### 7. Получение событий пользователя

Запрос:
SELECT e.*
FROM events e
JOIN event_participants ep ON e.id = ep.event_id
WHERE ep.user_id = 1;

Оптимизация:
Используется индекс по user_id.

Вывод:
Запрос выполняется быстрее за счёт уменьшения объёма данных. Здесь важен индекс по user_id в таблице event_participants, потому что именно он позволяет быстро найти все события, на которые зарегистрирован конкретный пользователь. После этого выполняется соединение с таблицей events. Такой запрос используется часто, поэтому тут нужен индекс по user_id.

### 8. Подсчёт количества участников

Запрос:
SELECT e.id, e.title, COUNT(ep.user_id)
FROM events e
LEFT JOIN event_participants ep ON e.id = ep.event_id
GROUP BY e.id, e.title;

Оптимизация:
Используется индекс по event_id.

Вывод:
Индекс ускоряет агрегацию и соединение таблиц. Оптимизация этого запроса  связана с индексом по event_id, так как по этому полю выполняется соединение таблиц. Фильтрация по статусу активной регистрации позволяет считать только актуальных участников, а не все записи подряд. Такой запрос можно использовать для получения аналитки по популярности событий.

Похожим образом работает запрос на количество событий у каждого организатора

Запрос:
SELECT u.id,
       u.login,
       u.first_name,
       u.last_name,
       COUNT(e.id) AS events_created
FROM users u
LEFT JOIN events e
       ON u.id = e.organizer_id
GROUP BY u.id, u.login, u.first_name, u.last_name
ORDER BY events_created DESC, u.id;

Вывод:
Здесь полезен индекс по organizer_id в таблице events, поскольку он ускоряет соединение между пользователями и их событиями. Этот запрос также можно отнести к аналитическим, так как он показывает активность организаторов.

### 9. Пользователи без регистраций

Запрос:
SELECT u.id, u.login
FROM users u
LEFT JOIN event_participants ep ON u.id = ep.user_id
WHERE ep.user_id IS NULL;

Оптимизация:
Индекс по user_id ускоряет LEFT JOIN.

Вывод:
Запрос можно использовать для аналитики. Здесь помогает индекс по user_id в таблице event_participants. Данный запрос интересен тем, что показывает не просто данные по событиям, а позволяет сделать вывод о вовлечённости пользователей.

### 10. Отмена регистрации

Запрос:
UPDATE event_participants
SET status = 'cancelled'
WHERE event_id = 1 AND user_id = 5;

Оптимизация:
Используется уникальный индекс (event_id, user_id).

Вывод:
Поиск записи выполняется быстро. В данном случае особенно полезно ограничение UNIQUE (event_id, user_id), так как оно не только защищает от повторной регистрации, но и позволяет быстро находить нужную запись при обновлении. То есть ограничения здесь одновременно работают и как механизм контроля целостности, и как элемент оптимизации.

## Итог

Если сравнивать базовые и оптимизированные запросы, то основная идея улучшения сводится к трём вещам. 
Во-первых, в оптимизированных запросах чаще выбираются только нужные поля вместо SELECT *. 
Во-вторых, фильтрация по status = 'registered' добавляется сразу в запросы, чтобы не обрабатывать лишние строки. 
В-третьих, структура JOIN-запросов строится так, чтобы сначала отбирать более узкий набор строк из таблицы event_participants, а потом уже соединять его с основной таблицей.

Наибольший эффект дают индексы по login, event_date, event_id и user_id, так как именно эти поля чаще всего участвуют в поиске и соединениях.

## Партиционирование

Партиционирование в рамках данной лабораторной работы не реализовывалось, так как объём тестовых данных небольшой. Однако для реальной системы можно было бы партиционировать таблицу events, птак как она содержит поле event_date, по которому часто выполняется фильтрация. В таком случае события можно было бы разбивать по месяцам или по годам. 
Например, таблицу `events` можно разделить по месяцам или по годам:
- `events_2026_04`
- `events_2026_05`
- `events_2026_06`

В этом случае при выполнении запросов вида:

```sql
SELECT * FROM events WHERE event_date = '2026-04-20';
PostgreSQL сможет обращаться не ко всей таблице, а только к нужному разделу, что уменьшит объём обрабатываемых данных.

Или можно попробовать партиционировать таблицу event_participants, если число регистраций станет очень большим. Таблица users для партиционирования подходит хуже, так как основные запросы к ней строятся по id, login и email, и для этих случаев обычно достаточно индексов.
