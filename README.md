Выбран **вариант №22 — Система управления событиями**.
Пример подобной системы:
https://www.eventbrite.com/
Такая система позволяет пользователям создавать события и регистрироваться на них.

# 3. Описание системы
Разрабатываемая система представляет собой **платформу управления событиями**.
Пользователь может:
* просматривать доступные события
* создавать собственные события
* регистрироваться на события
* просматривать список своих событий
* просматривать участников события
* отменять регистрацию на событие

Система может использоваться для организации различных мероприятий: конференций, митапов, лекций и других событий.

# 4. Основные сущности системы
## Пользователь (User)
Пользователь — это человек, зарегистрированный в системе.
Пример атрибутов:
* id
* login
* firstName
* lastName
* email

## Событие (Event)
Событие представляет собой мероприятие, которое создаётся пользователем системы.
Пример атрибутов:
* id
* title
* description
* date
* location
* organizerId

## Участник (Participant)
Участник — это пользователь, зарегистрированный на конкретное событие.
В архитектуре системы участник рассматривается как **запись о регистрации пользователя на событие**.
Пример атрибутов:
* userId
* eventId
* registrationDate
* status

# 5. Требуемый API
Система должна реализовывать следующий набор операций.

## Работа с пользователями
* создание нового пользователя
* поиск пользователя по логину
* поиск пользователя по маске имени и фамилии

## Работа с событиями
* создание события
* получение списка событий
* поиск событий по дате

## Работа с регистрациями
* регистрация пользователя на событие
* получение участников события
* получение событий пользователя
* отмена регистрации на событие

Эти операции реализуются в соответствующих микросервисах системы.

# 6. Роли пользователей
В системе выделена основная роль:

## Пользователь
Пользователь может:
* просматривать события
* создавать события
* регистрироваться на события
* просматривать свои события
* отменять регистрацию
* просматривать участников события

Отдельная роль **организатора** не выделяется, так как один и тот же пользователь может быть как участником событий, так и их создателем.

# 7. Внешние системы

Система взаимодействует со следующими внешними сервисами.

## Email-сервис
Используется для отправки уведомлений пользователям.
Например:
* подтверждение регистрации на событие
* уведомление об отмене регистрации

# 8. Архитектурный подход
Для системы выбрана **микросервисная архитектура**.
Каждый сервис отвечает за свою область функциональности.
Преимущества такого подхода:
* разделение ответственности между сервисами
* независимое масштабирование компонентов
* возможность использования различных технологий
* повышение отказоустойчивости системы

# 9. Контейнеры системы
В архитектуре системы выделены следующие контейнеры.

## Веб-приложение
Пользовательский интерфейс системы.
Через веб-приложение пользователь может:
* просматривать события
* создавать события
* регистрироваться на события
* просматривать свои события
* смотреть участников события
* отменять регистрацию

Технологии:
React
Web Browser

## API Gateway
Единая точка входа в backend-систему.
Основные задачи:
* маршрутизация запросов к микросервисам
* централизованная обработка ошибок
* возможная авторизация пользователей

## Сервис пользователей
Отвечает за управление пользователями.
Функции:
* создание пользователя
* поиск пользователя по логину
* поиск пользователя по имени и фамилии

## Сервис событий
Отвечает за управление событиями.
Функции:
* создание события
* получение списка событий
* поиск событий по дате

## Сервис регистраций
Отвечает за регистрацию пользователей на события.
Функции:
* регистрация пользователя на событие
* получение участников события
* получение событий пользователя
* отмена регистрации

Для проверки данных сервис регистраций взаимодействует с:
* сервисом пользователей
* сервисом событий

## Сервис уведомлений
Отвечает за отправку уведомлений пользователям.
Функции:
* отправка email-уведомлений
* отправка подтверждений регистрации

# 10. Хранилища данных
Каждый микросервис имеет собственное хранилище данных.

## База пользователей
Хранит данные пользователей.

Технология: **PostgreSQL**

## База событий
Хранит данные событий.

Технология: **PostgreSQL**

## База регистраций
Хранит информацию о регистрациях пользователей на события.

Технология: **PostgreSQL**

# 11. Дополнительные инфраструктурные компоненты
## NoSQL-хранилище
Используется для быстрых операций чтения и поисковых представлений событий.

Пример технологии: **MongoDB**

## Система кеширования
Используется для кеширования часто запрашиваемых данных.

Например:
* популярные события
* список событий
Пример технологии: **Redis**

## Брокер сообщений

Используется для асинхронного взаимодействия между сервисами.
Например:
* событие регистрации пользователя
* отправка уведомлений

Пример технологии: **RabbitMQ**

# 12. Основной архитектурный сценарий
В качестве архитектурно значимого сценария выбран:

**Регистрация пользователя на событие**

Последовательность действий:

1. Пользователь открывает страницу события
2. Пользователь нажимает кнопку регистрации
3. Веб-приложение отправляет запрос в API Gateway
4. API Gateway передает запрос в сервис регистраций
5. Сервис регистраций проверяет существование пользователя через сервис пользователей
6. Сервис регистраций проверяет событие через сервис событий
7. Создается запись регистрации в базе регистраций
8. Сервис регистраций публикует событие регистрации в брокер сообщений
9. Сервис уведомлений получает событие и отправляет email пользователю

Данный сценарий выбран, так как он затрагивает большинство компонентов системы.
