## Описание работы

В данной лабораторной работе реализована документная база данных MongoDB для системы управления событиями.  
Эта работа у меня идет как продолжение лабораторной работы №3, где я делала реляционную модель на PostgreSQL.

В ЛР3 использовались таблицы:

- `users`
- `events`
- `event_participants`

В ЛР4 я сохранила ту же предметную область, те же основные сущности и те же сценарии, но уже перенесла хранение данных на MongoDB.

Для совместимости с прошлой лабораторной работой в API я оставила числовые `id`, хотя внутреннее поле MongoDB `_id` тоже используется.

## Что реализовано
В проекте реализованы:
- проектирование документной модели;
- тестовые данные для MongoDB;
- CRUD-запросы для основных операций;
- валидация схемы через `$jsonSchema`;
- API для работы с MongoDB;
- Docker-конфигурация для запуска MongoDB и API.

В тестовых данных используются разные типы данных MongoDB:
- `String`
- `Number`
- `Date`
- `Array`
- `Object`

## Структура проекта
- `schema_design.md` — описание проектирования документной модели;
- `data.js` — заполнение тестовыми данными;
- `queries.js` — основные MongoDB-запросы;
- `validation.js` — настройка и проверка валидации;
- `app/` — FastAPI приложение;
- `tests/` — простые тесты API;
- `Dockerfile` — контейнер приложения;
- `docker-compose.yml` — запуск MongoDB и API.

## Используемые коллекции

В MongoDB используются три основные коллекции:
- `users`
- `events`
- `event_participants`


## Основные операции

Реализованы следующие операции:

- создание нового пользователя;
- поиск пользователя по логину;
- поиск пользователя по маске имени и фамилии;
- создание события;
- получение списка событий;
- поиск событий по дате;
- регистрация пользователя на событие;
- получение участников события;
- получение списка событий пользователя;
- отмена регистрации.

Также в `queries.js` добавлены аналитические запросы:
- количество участников на каждом событии;
- самые популярные события;
- количество событий у каждого организатора;
- города с наибольшим количеством событий.

## Почему API похоже на ЛР3
При переносе проекта на MongoDB я постаралась оставить API максимально близким к третьей лабораторной работе.  
Еще я сохранила числовые `id`, чтобы структура осталась похожей на SQL-версию и было проще переносить уже готовые сценарии.

## Как запустить проект

### 1. Запуск через Docker
```bash
docker compose up --build -d
```

После запуска будут доступны:

- API: `http://localhost:8001`
- Swagger UI: `http://localhost:8001/docs`
- MongoDB: `mongodb://localhost:27017`

### 2. Загрузка тестовых данных
```bash
docker compose exec mongo mongosh --file /scripts/data.js
```

### 3. Подключение валидации схемы
```bash
docker compose exec mongo mongosh --file /scripts/validation.js
```

### 4. Выполнение MongoDB-запросов
```bash
docker compose exec mongo mongosh --file /scripts/queries.js
```

## Локальный запуск API
Если запускать без Docker:
```bash
pip install -r requirements.txt
uvicorn app.main:app --reload
```

По умолчанию используются:

- `MONGO_URL=mongodb://localhost:27017/`
- `MONGO_DB_NAME=events_db`

При локальном запуске без Docker API будет доступно на `http://localhost:8000`.

## Тестирование
Для проверки API можно выполнить:
```bash
python3 -m pytest
```

Тесты используют `mongomock`, поэтому для них не требуется отдельный запущенный MongoDB сервер.
