В третьей лабораторной работе для этого варианта у меня уже была реализована реляционная база данных PostgreSQL. В ней использовались три основные таблицы:

## Какие сущности нужно хранить

Для системы управления событиями нужно хранить:

- пользователей;
- события;
- регистрации пользователей на события.

В MongoDB я также оставляю три основные коллекции:

- `users`
- `events`
- `event_participants`

Я выбрала такой вариант, потому что он получился самым понятным и не требует заново перестраивать всю систему.

## Коллекция users

В коллекции `users` хранятся пользователи системы.

Пример документа:

```json
{
  "_id": ObjectId("..."),
  "id": 1,
  "login": "maria_lemeshchenko",
  "first_name": "Мария",
  "last_name": "Лемещенко",
  "email": "marialemeshchenko@mail.ru",
  "password_hash": "hash_privet_mir1",
  "roles": ["participant"],
  "profile": {
    "city": "Москва",
    "phone": "+79990000001"
  },
  "created_at": ISODate("2026-04-01T10:00:00Z")
}
```

### Почему пользователь хранится отдельно

Пользователь здесь является отдельной сущностью:

- он участвует в создании событий;
- он может регистрироваться на разные события;
- его данные используются в нескольких сценариях API.

Поэтому хранить пользователя прямо внутри других документов мне показалось неудобно, потому что данные начали бы дублироваться.

При этом для демонстрации возможностей MongoDB я добавила в пользователя небольшие вложенные данные:

- `roles` как массив;
- `profile` как embedded document.

Эти поля не влияют на основные связи системы, но позволяют использовать в модели не только строки и числа.

## Коллекция events

В коллекции `events` хранятся события.

Пример документа:

```json
{
  "_id": ObjectId("..."),
  "id": 1,
  "title": "Архитектурный митап",
  "description": "Встреча по архитектуре приложений",
  "event_date": ISODate("2026-04-01T00:00:00Z"),
  "location": "Москва",
  "organizer_id": 1,
  "tags": ["architecture", "meetup"],
  "venue": {
    "hall": "Аудитория 305",
    "format": "offline"
  },
  "created_at": ISODate("2026-04-01T10:30:00Z")
}
```

### Почему organizer_id хранится как reference

В реляционной модели ЛР3 поле `organizer_id` было внешним ключом на таблицу `users`.  
В MongoDB я оставила ту же идею и храню организатора по ссылке:

- организатор может создавать много событий;
- данные организатора не нужно дублировать в каждом событии;
- при изменении данных пользователя не придется изменять все связанные события.

Поэтому здесь мне показалось логичнее оставить reference.

Дополнительно в событиях я использую:

- `tags` как массив;
- `venue` как небольшой вложенный объект.

То есть основные связи у меня сделаны через references, а небольшие зависимые данные можно хранить внутри документа события.

## Коллекция event_participants

В коллекции `event_participants` хранятся регистрации пользователей на события.

Пример документа:

```json
{
  "_id": ObjectId("..."),
  "id": 1,
  "event_id": 1,
  "user_id": 2,
  "registration_date": ISODate("2026-04-02T09:00:00Z"),
  "status": "registered"
}
```

## Почему регистрация вынесена в отдельную коллекцию

В ЛР3 таблица `event_participants` связывала пользователей и события.  
В MongoDB я тоже оставила отдельную коллекцию и не стала встраивать участников прямо в документ события.

Причины у меня были такие:

- один пользователь может участвовать во многих событиях;
- у одного события может быть много участников;
- связь имеет собственные поля, например `registration_date` и `status`;
- нужно удобно получать как участников события, так и список событий конкретного пользователя.

По сути здесь остается связь many-to-many, поэтому отдельная коллекция для регистраций смотрится нормально и по смыслу, и по структуре.

## Где используются embedded documents

В этой работе основную логику я строю на references, потому что структура уже пришла из ЛР3 и для нее удобнее оставить три отдельные коллекции.  
Но полностью от embedded documents я не отказалась. У меня они есть в небольших полях:

- `users.profile`
- `users.roles`
- `events.tags`
- `events.venue`

То есть связи между сущностями я храню отдельно, а маленькие зависимые данные оставляю внутри документа.

## Итог по выбору embedded и references

- `users` хранится отдельно;
- `events` хранится отдельно;
- `event_participants` хранится отдельно;
- `events.organizer_id` — reference на `users`;
- `event_participants.user_id` — reference на `users`;
- `event_participants.event_id` — reference на `events`.
- `users.profile` — embedded object;
- `users.roles` — embedded array;
- `events.tags` — embedded array;
- `events.venue` — embedded object.

Такой подход хорошо сохраняет структуру предыдущей лабораторной работы и при этом нормально переносится на MongoDB.

## Индексы

Для ускорения основных запросов в MongoDB я ориентировалась на индексы, которые уже были полезны в ЛР3:

- уникальный индекс по `users.login`;
- уникальный индекс по `users.email`;
- индекс по `users.first_name`;
- индекс по `users.last_name`;
- индекс по `events.event_date`;
- индекс по `events.organizer_id`;
- индекс по `event_participants.event_id`;
- индекс по `event_participants.user_id`;
- составной уникальный индекс по `event_participants.event_id` и `event_participants.user_id`.

Последний индекс для меня был самым важным, потому что он не дает зарегистрировать одного и того же пользователя на одно и то же событие два раза.

Отдельно отмечу, что в рабочем API я оставила числовые `id`, чтобы структура осталась ближе к ЛР3. При этом внутреннее поле MongoDB `_id` у документов тоже остается.
