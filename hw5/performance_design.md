## Связь с предыдущей работой
Лабораторная работа №5 выполнена на основе ЛР4.  
В предыдущей работе был реализован REST API для системы управления событиями на базе MongoDB.  
В рамках ЛР5 предметная область не изменялась, а основное внимание было уделено вопросам производительности:

- кеширование;
- rate limiting;
- анализ влияния этих механизмов на работу приложения.

## 1. Анализ производительности
### Какие операции выполняются часто
Для рассматриваемой предметной области наиболее часто используются следующие операции чтения:

- список всех событий;
- события по выбранной дате;
- участники конкретного события;
- события текущего пользователя.

Перечисленные запросы можно отнести к основным hot paths, так как они ориентированы на чтение данных и потенциально могут вызываться многократно.

### Какие операции могут быть медленными
Наиболее затратными по времени в данном проекте являются:

- чтение списка событий из MongoDB;
- выборка участников события;
- получение событий пользователя через коллекцию `event_participants`;
- частые запросы на авторизацию и поиск пользователей.

Особенно это проявляется в ситуации, когда одни и те же данные запрашиваются повторно, и приложение каждый раз обращается к базе данных без промежуточного хранения результата.

### Требования к производительности
Для учебного проекта были приняты следующие ориентиры:

- часто вызываемые GET-запросы должны обрабатываться быстрее за счет повторного использования ранее полученных данных;
- повторный запрос к уже кешированным данным должен выполняться без повторного обращения к MongoDB;
- сервис не должен позволять бесконтрольно отправлять чувствительные запросы, например запросы на вход;
- при превышении лимита API должен возвращать `429 Too Many Requests`.

## 2. Стратегия кеширования
### Выбор данных для кеширования
Для кеширования были выбраны данные, которые:

- часто читаются;
- не изменяются при каждом обращении;
- могут храниться в кеше ограниченное время без нарушения логики работы приложения.

В итоге кешируются:

1. список всех событий;
2. список событий по дате;
3. список участников события;
4. список событий пользователя.

### Вбранная стратегия
Для всех перечисленных запросов используется стратегия **Cache-Aside**.

Общая схема работы выглядит следующим образом:

1. API проверяет наличие подготовленного ответа в кеше;
2. при наличии кеша ответ возвращается без обращения к базе данных;
3. при отсутствии кеша данные читаются из MongoDB;
4. после этого результат сохраняется в Redis на ограниченное время.

Данный подход был выбран как наиболее понятный и удобный для учебного REST API.

### TTL для кеша
Для разных запросов использованы следующие значения TTL:

- список всех событий — `60` секунд;
- события по дате — `90` секунд;
- участники события — `45` секунд;
- события пользователя — `45` секунд.

Для списка событий по дате был выбран немного больший TTL, так как такие данные обычно не изменяются слишком часто.

### Инвалидация кеша
Инвалидация кеша выполняется в точках изменения данных:

- при создании события очищается кеш со списками событий;
- при регистрации на событие очищается кеш участников события и кеш событий пользователя;
- при отмене регистрации очищается тот же набор кешей.

Схема инвалидации намеренно оставлена простой, чтобы сохранить понятную и предсказуемую логику обновления данных.

## 3. Реализация кеширования
Для кеширования в проекте используется Redis.

Согласно требованиям задания кеширование реализовано минимум для двух endpoints:

- `GET /api/v1/events`
- `GET /api/v1/events/by-date`

Дополнительно кешируются:

- `GET /api/v1/events/{event_id}/participants`
- `GET /api/v1/users/me/events`

Для наглядной проверки в ответ был добавлен заголовок:

- `X-Cache: HIT`
- `X-Cache: MISS`

Это позволяет определить, был ли ответ получен из кеша или сформирован заново на основе данных из базы.

## 4. Проектирование rate limiting
### Выбор endpoints для ограничения
Ограничение частоты запросов добавлено для следующих endpoints:

- `POST /api/v1/auth/login`
- `GET /api/v1/users/search`

Выбор связан со следующими причинами:

- запрос на логин является чувствительным и может использоваться слишком часто при попытках перебора;
- поиск пользователей также может вызываться многократно и создавать лишнюю нагрузку.

### Выбранный алгоритм
Для реализации rate limiting был выбран алгоритм **Fixed Window Counter**.

Этот вариант подходит для учебного проекта, так как он достаточно прост в реализации и при этом понятен с точки зрения принципа работы.

### Используемые лимиты
В проекте заданы следующие ограничения:

- логин — `5` запросов в `60` секунд с одного клиента;
- поиск пользователей — `30` запросов в `60` секунд с одного клиента.

## 5. Реализация rate limiting
Rate limiting реализован с использованием Redis, где хранятся счетчики запросов.

При превышении лимита сервис возвращает:

- HTTP `429 Too Many Requests`

Кроме того, в ответ включаются заголовки:

- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `X-RateLimit-Reset`

За счет этого клиент может определить:

- установленный лимит;
- сколько запросов осталось;
- когда лимит сбросится.

## 6. Как это влияет на производительность
### Что дает кеширование
Кеширование позволяет уменьшить число повторных обращений к MongoDB.  
Если данные уже были недавно запрошены, API может вернуть готовый ответ из Redis без повторного выполнения того же запроса к базе.

В результате:

- снижается нагрузка на базу данных;
- быстрее отдаются часто читаемые запросы;
- повторные GET-запросы выполняются более стабильно.

### Что дает rate limiting
Rate limiting не ускоряет отдельный запрос напрямую, однако защищает систему от чрезмерно частого обращения к отдельным endpoints.

За счет этого:

- приложение не так просто перегрузить частыми однотипными запросами;
- уменьшается риск перебора логинов;
- нагрузка становится более контролируемой.

### Какие метрики можно смотреть
Для оценки результата можно использовать следующие метрики:

- среднее время ответа;
- число запросов в минуту;
- число ответов `429`;
- количество обращений к MongoDB;
- cache hit rate.

### Как измерить эффективность кеша
Эффективность кеширования можно оценивать по показателю hit rate:

`hit rate = cache_hits / (cache_hits + cache_misses)`

Чем выше доля попаданий в кеш, тем чаще повторные запросы обслуживаются без повторного чтения данных из MongoDB.

## Вывод
В данной лабораторной работе API из ЛР4 был дополнен двумя механизмами оптимизации:

- кеширование через Redis;
- rate limiting через Fixed Window Counter.

В результате удалось показать базовые подходы к оптимизации производительности, которые применяются в backend-приложениях: уменьшение числа повторных обращений к базе данных и ограничение чрезмерно частых запросов к отдельным endpoints.
